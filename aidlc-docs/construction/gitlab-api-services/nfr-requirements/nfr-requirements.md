# UNIT-02: GitLab API Services NFR要件

## 概要

UNIT-02（GitLab API Services）の非機能要件を定義する。ローカルCLI/TUIツールのAPI層として、パフォーマンス、セキュリティ、信頼性、保守性の観点で要件を規定する。

---

## 1. パフォーマンス要件

### NFR-PERF-01: APIタイムアウト

- **接続タイムアウト**: 10秒 — GitLabサーバーへのTCP接続確立の上限
- **読み取りタイムアウト**: 30秒 — APIレスポンス受信の上限
- python-gitlabの `timeout` パラメータで設定する
- タイムアウト発生時は `GitLabConnectionError` を送出する

### NFR-PERF-02: 非同期API呼び出し

- すべてのAPI呼び出しメソッド��� **async/await** で実装する
- python-gitlabは同期ライブラリのため、`asyncio.to_thread()` でブロッキング呼び出しをラップする
- TUI層から直接 `await` で呼び出し可能にする
- UIスレッドをブロックしないことを保証する

### NFR-PERF-03: キャッシュメモリ制限

- セッション内メモリキャッシュにエントリ数上限を設定する：

| キャッシュ対象 | 上限エントリ数 | キー |
|---|---|---|
| MR詳細（MergeRequestDetail） | 100件 | mr_iid |
| 変更ファイル一覧（list[FileChange]） | 100件 | mr_iid |
| ファイル差分（FileDiff） | 50件 | (mr_iid, file_path) |
| ディスカッション（list[Discussion]） | 100件 | mr_iid |

- 上限超過時はLRU（Least Recently Used）方式で古いエントリを破棄する
- MR一覧はキャッシュ対象外（常にAPI呼び出し）

### NFR-PERF-04: MR一覧初回取得

- NFR-05（要件定義）に基づき、MR一覧の初回取得は5秒以内を目標とする
- 4カテゴリのAPI呼び出しを並行実行（`asyncio.gather`）して合計待ち時間を短縮する

---

## 2. セキュリティ要件

### NFR-SEC-01: トークンの取り扱い（SECURITY-03準拠）

- python-gitlabインスタンス生成時のみトークンを使用する
- トークンをログ出力に含めない
- トークンをエラーメッセージに含めない
- APIリクエストヘッダのログ出力を禁止する

### NFR-SEC-02: エラーレスポンスの情報漏洩防止（SECURITY-09準拠）

- python-gitlabの例外メッセージをそのままユーザーに表示しない
- ユーザー向けメッセージは事前定義の汎用メッセージを使用する
- 詳細なエラー情報（HTTPステータス、レスポンスボディ、API URL）はログファイルにのみ記録する

### NFR-SEC-03: 外部API呼び出しのエラーハンドリング（SECURITY-15準拠）

- すべてのpython-gitlab API呼び出しをtry-exceptで保護する
- 予期しない例外（KeyError、AttributeError等）もキャッチ���、汎用エラーとして処理する
- 例外のスタックトレースはログファイル��のみ記録する

### NFR-SEC-04: SSL/TLS検証

- python-gitlabのSSL検証をデフォルトで有効にする（`ssl_verify=True`）
- セルフホスト環境でカスタムCA証明書が必要な場合は設定で指定可能にする

## 3. 信頼性要件

### NFR-REL-01: API接続失敗時の挙動

- リトライは行わない（機能設計Q3: A）
- 接続失敗時は即座にユーザーにエラーメッセージを表示する
- アプリケーション���クラッシュせず、エラー状態をTUI上に表示する

### NFR-REL-02: 不正なAPIレスポンスへの対応

- python-gitlabオブジェクトから内部モデルへの変換時に、期待するフィールドが欠損している場合はデフォルト値を使用する
- 変換不能な場合はログに��録し、該当データをスキップする（一覧全体のエラーにしない）

### NFR-REL-03: セッション中の接続切断

- API��び出し時に接続エラーが発生した場合、エラーメッセージを表示する
- キャッシュ済みデータは保持し、再接続後に使用可能とする
- ユーザーの明示的リフレッシュ操作で再接続を試みる

---

## 4. 保守性要件

### NFR-MAINT-01: テスト戦略

- **単体テスト（モック）**: python-gitlabの呼び出しをモックし、以下を検証する
  - python-gitlabオブジェクトから内部モデルへの変換ロジック
  - ビジネスルール（フィルタリング、バリデーション）
  - エラーハンドリング��各例外パターン）
  - キャッシ��の動作（ヒット/ミス/無効化/LRU破棄）
- **統合テスト（実環境）**: 実際のGitLabインスタンスに接続し、以下を検証する
  - 認証フロー
  - MR一覧・詳細・差分の取得
  - コメント投稿・リプライ
  - ※CIでは環境変数でGitLabトークンを提供、ローカルでは`pytest.mark.integration`でスキップ可能

### NFR-MAINT-02: コードカバレッジ

- 単体テストのカバレッジ目標: 80%以上
- 変換ロジックとエラーハンドリングは100%カバレッジを目指す

### NFR-MAINT-03: ログ出力

- UNIT-01のLogger（構造化ログ）を使用する
- API呼び出しのログレベル:
  - DEBUG: API呼び出しの開始・完了（URL、パラメータ ※トークン除外）
  - WARNING: 予期しないレスポンス形式
  - ERROR: API接続失敗、認証失敗
